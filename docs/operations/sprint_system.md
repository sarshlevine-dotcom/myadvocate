# Sprint System — MyAdvocate

**Status:** Active  
**Last updated:** 2026-03-21

---

## Purpose

Defines how work moves from idea → implementation → validation → deployment.

---

## Core model

Work is executed in **tight, outcome-focused sprints**.

Each sprint should:
- ship something real
- reduce risk
- move toward launch gates

---

## Sprint structure

### Inputs
- backlog (Notion / repo issues)
- system constraints (SYSTEM.md)
- current bottlenecks

### Execution
- Claude/Cowork builds
- repo updated
- docs updated

### Validation
- tests run
- compliance check
- manual review where required

### Output
- deployed feature
- updated docs
- exportable summary

---

## Sprint outputs

Each sprint should produce:
- working code
- updated documentation
- optional export doc for Drive

---

## Feedback loop

1. Deploy
2. Observe
3. Analyze (NotebookLM / metrics)
4. Adjust
5. Next sprint

---

## Key rules

- no orphan work
- no undocumented systems
- no features without defined purpose

---

## Relationship to system

Sprints are the engine that keeps:
- product evolving
- docs updated
- exports flowing
- analysis improving

---

## Critical insight

Speed comes from clarity and system alignment, not raw output volume.
